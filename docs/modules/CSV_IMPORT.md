# CSV Import Module

The **CSV Import Module** (`src/modules/csv-import`) provides a robust, scaleable solution for bulk-importing user data into the dynamic entities generated by AppForge AI.

## 🏗 Architecture & Chunking Strategy

A major challenge with Serverless environments (like Vercel) is the strict execution timeout (10 to 60 seconds). Parsing and inserting a 10,000-row CSV file entirely on the server within a single API request will frequently timeout.

To resolve this, the module uses a **Client-Side Chunking Pattern**:
1. **PapaParse (Client)**: The browser reads the CSV file and parses it into JSON without uploading the raw file to the server.
2. **Batching**: The frontend groups the JSON rows into chunks (default 100 rows per chunk).
3. **Server Actions (`import.actions.ts`)**: The frontend sends chunks sequentially to the backend.
4. **Progress UI**: The client displays a live progress bar based on the HTTP responses of each chunk.

## 📂 Structure

- `components/CSVImportWizard.tsx`: The main orchestration component managing the 4 steps.
- `components/steps/Step1Upload.tsx`: Drag-and-drop file upload and PapaParse initialization.
- `components/steps/Step2Map.tsx`: Fuzzy auto-matching of CSV Headers to Entity Fields.
- `components/steps/Step3Preview.tsx`: Client-side Zod validation against the first 10 rows.
- `components/steps/Step4Progress.tsx`: The chunk execution engine and live progress reporter.
- `services/import-job.service.ts`: The backend logic that hooks into `CRUDEngine` to perform the actual database insertions and validations.

## 🛡 Validation & Audit

Because the `ImportJobService` internally delegates row insertions to `CRUDEngine.create()`, the import process automatically inherits all security policies from the Backend Runtime Engine (Phase 5):
- **Zod Validation**: Every single row is validated against the dynamic schema. If a chunk fails, it does not crash the whole job. Instead, the specific row error is caught and appended to the `CSVImportJob.errorLog`.
- **Audit Logs**: Every successful insertion automatically writes an `AuditLog` entry, tracking the data lineage back to the import process.

## 🔄 Failure Handling (Partial Success)

If a CSV contains 1,000 rows and 5 of them have invalid email addresses:
1. 995 rows will be successfully imported (`successRows = 995`).
2. 5 rows will fail (`errorRows = 5`).
3. The job status will be marked as `FAILED_PARTIAL`.
4. The user can view the specific rows and the reason they failed in the Error Log UI.
