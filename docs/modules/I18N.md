# Internationalization (i18n) Module

The **i18n Module** (`src/modules/i18n`) provides multi-language capabilities for AppForge AI using Next.js 15 App Router standard practices.

## 🏗 Architecture

AppForge uses **Route Localization** to manage languages.
This means the locale code is injected directly into the URL path:
- `/en/dashboard`
- `/hi/dashboard`
- `/kn/dashboard`

### 1. Middleware (`src/middleware.ts`)
The Next.js middleware intercepts every incoming request. If a user visits `/dashboard` without a locale prefix, the middleware automatically redirects them to `/en/dashboard` (or their preferred language based on cookies/headers).

### 2. Dictionaries (`src/modules/i18n/dictionaries`)
Static JSON files provide the translations.
- `en.json` (English - Fallback & Base Type Definition)
- `hi.json` (Hindi)
- `kn.json` (Kannada)

*Note: The `i18n.types.ts` strictly infers the `Dictionary` interface from `en.json`. If a developer tries to use a key that doesn't exist in English, TypeScript will throw an error.*

### 3. Dynamic Loading (`i18n.service.ts`)
We use React Server Components to our advantage. The `getDictionary(locale)` function dynamically `import()`s the requested JSON file on the server.
This guarantees that if a user visits `/en/dashboard`, the `hi.json` and `kn.json` files are never parsed by the server or sent to the client bundle.

### 4. Language Switcher (`LanguageSwitcher.tsx`)
A client-side UI component that allows users to seamlessly switch languages. It reads the current pathname (`usePathname`), swaps the locale segment, and pushes the new route via the Next.js `useRouter`.
